# 账号按数量分文件夹保存设计

日期：2026-06-07

## 背景

当前生成成功的账号统一保存到输出目录下的 `accounts.json`。新增需求是在生成账号时支持配置“每成功生成几个账号创建一个文件夹保存”，便于用户按批次管理结果文件。

## 目标

新增一个可配置项 `accountsPerFolder`：

- 默认值为 `0`，表示关闭分文件夹，保持当前行为。
- 当值为 `N > 0` 时，每成功生成 `N` 个账号切换到下一个批次文件夹。
- 分组依据为成功账号数量，失败任务不计入。

## 保存规则

当 `accountsPerFolder` 未配置、为空、为 `0` 或非法值时，账号仍保存到当前输出目录：

```text
<output>/accounts.json
```

当 `accountsPerFolder = N` 时：

```text
<output>/batch-001/accounts.json  # 第 1..N 个成功账号
<output>/batch-002/accounts.json  # 第 N+1..2N 个成功账号
<output>/batch-003/accounts.json  # 以此类推
```

批次编号从 `001` 开始，使用三位补零格式，便于文件夹自然排序。

## 架构设计

### 前端

在现有配置表单中新增数字输入项，用于设置每个文件夹保存的成功账号数量。

前端职责：

- 从 `localStorage['kiro-config']` 读取该配置。
- 保存设置时写回该配置。
- 启动任务时将该值放入任务请求对象。
- 对输入值做基础解析，空值或非法值传 `0`。

### 后端任务请求

在现有 `StartTaskRequest` 中新增字段：

```go
AccountsPerFolder int `json:"accountsPerFolder"`
```

该字段只影响成功账号保存路径，不影响注册流程、邮箱逻辑、失败日志或输出目录选择。

### 后端保存流程

任务协调器在账号成功后维护成功计数：

1. 成功生成账号后递增成功计数。
2. 若 `AccountsPerFolder <= 0`，使用原始输出目录。
3. 若 `AccountsPerFolder > 0`，按成功计数计算批次编号：
   - `batchIndex = (successCount - 1) / AccountsPerFolder + 1`
   - 子目录名为 `batch-%03d`
4. 调用现有 `SaveKiroSuccess(result, targetDir)` 保存账号。

保存函数继续负责创建目录、读取/更新 `accounts.json`、按邮箱去重等既有逻辑。

## 并发与一致性

账号生成支持并发，因此成功计数必须在保存前以线程安全方式递增。推荐在任务协调器中使用互斥锁保护“成功计数递增 + 目标目录计算”这一临界区，避免多个成功结果被分到错误批次。

`SaveKiroSuccess` 的既有文件写入行为保持不变。本次需求只改变传入的目标目录，不扩展文件格式。

## 错误处理

- 配置值小于等于 `0`：视为关闭分文件夹。
- 配置值无法解析：前端传 `0`，后端也按关闭处理。
- 子目录创建或文件写入失败：沿用现有保存失败日志与错误处理路径。

## 测试重点

- 默认配置下仍写入 `<output>/accounts.json`。
- `accountsPerFolder = 2` 时，第 1、2 个成功账号写入 `batch-001`，第 3、4 个写入 `batch-002`。
- 失败任务不计入批次容量。
- 并发成功时批次分配不重复、不跳号。
- 前端配置保存、加载、启动任务传参正确。

## 非目标

- 不改变账号 JSON 数据结构。
- 不改变失败日志保存规则。
- 不新增按邮箱服务商或任务序号分组。
- 不迁移已有结果文件。
